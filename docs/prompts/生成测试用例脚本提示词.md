## 📤 O - 输出要求

### 1. 输出结构

#### 文件头部
- 导入必要的Playwright测试模块
- 导入相关的POM类
- 设置测试套件基本信息

#### 测试用例结构
- 使用describe块组织相关测试用例
- 每个test块代表一个完整的业务场景
- 合理的测试用例命名和描述

#### 测试步骤实现
- 页面对象实例化
- 按业务流程顺序调用POM方法
- 在关键节点添加断言验证
- 包含适当的控制台日志输出

### 2. 质量要求

#### 断言策略
- 使用Playwright的expect API进行状态验证
- 断言应该验证用户可感知的状态变化
- 避免过度断言导致测试脆弱性

#### 测试数据管理
- 测试数据应该参数化，避免硬编码
- 考虑测试数据的独立性和可重复性
- 敏感数据（如密码）使用环境变量或配置文件

#### 代码可维护性
- 遵循DRY原则，避免重复代码
- 使用清晰的变量命名和函数结构
- 添加必要的注释说明复杂逻辑

### 3. 格式规范

- **文件格式**: TypeScript (.spec.ts)
- **代码风格**: 遵循Playwright官方推荐规范
- **缩进**: 2个空格
- **命名规范**: 测试用例使用描述性命名

### 4. 特别说明

#### 测试组织策略
- 每个测试文件专注一个主要业务流程
- 使用beforeEach/afterEach处理通用的准备和清理逻辑
- 考虑测试用例之间的独立性

#### 错误处理和调试
- 在关键步骤添加console.log输出，便于调试
- 使用try-catch处理可能的异常情况
- 考虑测试失败时的截图和日志收集

#### 性能和稳定性
- 避免使用固定的等待时间（sleep）
- 充分利用Playwright的自动等待机制
- 考虑网络延迟和页面加载时间的影响

import { LoginPage } from '../pages/LoginPage';

    console.log('步骤2：用户登录');
    await loginPage.login('standard_user', 'secret_sauce');
    await expect(inventoryPage.pageTitle).toHaveText('Products');
    
    // 更多测试步骤...
  });
});
