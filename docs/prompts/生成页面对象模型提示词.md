## 🎯 G - 目标与意图

### 核心目标

生成符合企业级标准的Playwright POM类，确保元素定位稳定可靠，代码结构清晰可维护。

### 具体目标

1. **定位策略优化**: 优先使用data-test属性，其次使用ID，最后选择最稳健的CSS选择器
2. **代码结构规范**: 生成符合TypeScript规范的类结构，包含必要的类型声明
3. **方法完整性**: 自动识别页面核心交互功能，生成对应的操作方法
4. **可维护性保障**: 确保生成的代码易于理解、修改和扩展

### 业务价值

- **为测试开发者**: 快速生成高质量POM代码，避免手动编写重复的定位器代码
- **为项目维护**: 提供标准化的页面对象结构，降低测试代码维护成本
- **为团队协作**: 统一POM编写规范，提高代码可读性和团队开发效率

### 成功标准

- ✅ 生成的类包含所有关键页面元素的定位器
- ✅ 定位策略遵循最佳实践（data-test > ID > 稳健的CSS选择器）
- ✅ 包含页面核心交互方法（点击、输入、跳转等）
- ✅ 代码符合TypeScript类型安全规范

## 📤 O - 输出要求

### 1. 输出结构

#### 类声明部分
- 导入必要的Playwright类型（Page, Locator）
- 类名使用PascalCase命名规范
- 构造函数接收Page对象参数

#### 元素定位器部分
- 所有交互元素的Locator声明
- 使用readonly修饰符确保不可变性
- 合理的变量命名（驼峰式，语义化）

#### 方法定义部分
- 页面核心操作方法（goto、fill、click等）
- 异步方法使用async/await语法
- 方法参数使用适当的TypeScript类型

### 2. 质量要求

#### 定位策略优先级
- 第一优先级：`[data-test="xxx"]` 属性选择器
- 第二优先级：`#id` ID选择器
- 第三优先级：稳健的CSS选择器（避免依赖样式类名）

#### 代码规范性
- 使用TypeScript严格类型检查
- 变量命名语义化，避免简写和缩写
- 方法命名动词开头，体现操作意图

#### 可维护性要求
- 代码结构清晰，逻辑分层明确
- 避免硬编码，使用参数化设计
- 添加必要的JSDoc注释

### 3. 格式规范

- **文档格式**: TypeScript (.ts)
- **缩进**: 2个空格
- **引号**: 单引号
- **分号**: 必须添加

### 4. 特别说明

#### 元素识别策略
如果HTML中缺少data-test属性，你应该：
1. **优先推荐**: 建议开发团队添加data-test属性
2. **备选方案**: 使用最稳定的现有属性（如ID、role、type等）
3. **避免使用**: 样式相关的class名称或深层嵌套选择器

#### 命名规范
- 页面类名：`{PageName}Page`格式（如LoginPage）
- 元素变量：描述元素功能，以Input、Button、Link等结尾
- 方法名：动词开头，描述具体操作（如login、fillForm、clickSubmit）

### 5. 输出格式示例

```typescript
import { type Locator, type Page } from '@playwright/test';

export class {PageName}Page {
  readonly page: Page;
  readonly {element1}: Locator;
  readonly {element2}: Locator;

  constructor(page: Page) {
    this.page = page;
    this.{element1} = page.locator('[data-test="{test-id}"]');
    this.{element2} = page.locator('#{id}');
  }

  async {method1}() {
    // 实现具体操作
  }

  async {method2}(param: string) {
    // 实现参数化操作
  }
}
```
