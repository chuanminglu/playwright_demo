- E2E测试转化与自动化测试脚本设计

**特别专长**：

## 📋 T - 任务描述

基于提供的**用户故事**和**验收标准/验收规则清单**，生成完整的功能测试用例清单，包含详细的测试步骤、验证点和预期结果。

### 输入材料

#### 材料1：用户故事

```
{请粘贴需要生成测试用例的用户故事：}

【故事ID】US001
【故事标题】用户注册功能
【用户故事】作为新用户，我希望通过手机号注册账号，以便使用系统功能
【验收标准】
- 用户输入有效手机号和验证码后，可成功注册
- 手机号必须是11位数字
- 验证码60秒内有效
- 密码长度8-20位，包含字母和数字
```

#### 材料2：验收规则清单（可选）

```
{如果已经生成了验收规则清单，请粘贴：}

【功能规则】
F1: 用户输入有效手机号和验证码后，可成功注册
F2: 用户点击"发送验证码"按钮后，系统发送短信验证码

【数据规则】
D1: 手机号必须是11位数字
D2: 手机号不能重复注册
D3: 密码长度8-20位，包含字母和数字

【异常规则】
E1: 输入错误验证码时，显示"验证码错误"提示
E2: 验证码过期后，提示"验证码已过期，请重新获取"

【性能规则】
P1: 注册接口响应时间<2秒（正常网络）
```

#### 材料3：业务规则与系统信息

```
【系统名称】UUV统一认证平台
【功能模块】认证管理-用户管理-新增用户
【业务约束】
- 测试环境：测试服务器（https://test.example.com）
- 测试数据：使用测试环境专用手机号段（138开头）
```

### 任务上下文

本任务是从需求到测试的关键环节，将用户故事和验收标准转化为可执行的功能测试用例。测试用例需要覆盖正常流程、异常场景、边界值和参数验证，确保功能交付符合业务预期，并为未来的E2E自动化测试奠定基础。

## 🎯 G - 目标与意图

### 核心目标

生成完整、可执行、可追溯的功能测试用例清单，确保每个用户故事的功能交付质量，并支持手动测试和自动化测试转化。

### 具体目标

1. **测试覆盖完整性**: 覆盖正常路径、异常场景、边界值、参数验证，确保测试无遗漏
2. **操作可执行性**: 测试步骤详细具体，任何测试人员都能独立执行
3. **验证点明确性**: 预期结果量化表达，避免"正常""成功"等模糊词
4. **E2E转化友好性**: 测试步骤清晰可自动化，支持未来转化为Playwright等E2E测试脚本

### 业务价值

- **为QA团队**: 提供标准化的测试用例模板，提高测试执行效率和一致性
- **为开发团队**: 明确功能验收标准，减少理解偏差和返工成本
- **为产品验收**: 提供结构化验收依据，快速确认功能完整性
- **为自动化测试**: 作为E2E测试脚本的需求输入，降低自动化测试设计成本

### 成功标准

- ✅ 每个用户故事生成5-15个测试用例（根据复杂度）
- ✅ 覆盖所有验收规则和业务约束
- ✅ 每个测试用例可独立执行，无依赖性问题
- ✅ 验证点量化明确，可判定通过/失败
- ✅ 支持优先级分级（P0核心流程/P1重要功能/P2辅助功能）
- ✅ 至少50%的测试用例可转化为E2E自动化测试

## 📤 O - 输出要求

### 1. 输出结构

生成完整的功能测试用例表格，必须包含以下字段：

#### 测试用例表格字段说明

| 字段名 | 说明 | 示例 |
|--------|------|------|
| **用例编号** | 格式：`{系统简称}-{功能模块}-{序号}`<br>确保唯一性和可追溯性 | `UUV-USER-ADD-001` |
| **系统** | 被测系统的完整名称 | `UUV统一认证平台` |
| **功能模块** | 功能所属模块路径，使用"-"分隔 | `认证管理-用户管理-新增用户` |
| **用例概述** | 测试用例的简短描述（10-20字） | `异常测试（等价类：有效）` |
| **优先级** | P0=核心流程（阻塞性）<br>P1=重要功能<br>P2=辅助功能 | `高` 或 `P0` |
| **标签** | 测试类型标签，多个用逗号分隔 | `功能测试,正常流程` |
| **前提条件** | 执行测试前需要满足的条件 | `1. 管理员已登录<br>2. 测试用户不存在` |
| **输入数据或操作** | 详细的测试步骤，使用编号列表 | `1. 点击【新增用户】<br>2. 输入用户名"zhangssan"<br>3. 输入身份证"440101199001011234"` |
| **测试空白** | 预留列（可用于测试环境、依赖服务等备注） | 可留空或填写特殊说明 |
| **输出数据或操作** | 系统响应的状态变化或数据输出 | `1. 管理员已登录<br>2. 新增用户表单已打开<br>3. 用户名"zhangssan"未存在` |
| **预期结果** | 量化、可验证的预期结果，使用编号列表 | `1. 弹出新增用户表单<br>2. 用户名输入框显示"zhangssan"<br>3. 提示"用户创建成功"<br>4. 用户列表显示新用户` |
| **测试结果** | 测试执行结果 | `待测试` / `通过` / `失败` |

### 2. 质量要求

#### 可追溯性（强制）

- 每个测试用例必须追溯到验收规则或用户故事
- 在用例编号或备注中标注来源：`（来源：验收规则F1）` 或 `（来源：US001验收标准第2条）`
- 确保验收规则100%被测试用例覆盖

#### 操作可执行性（强制）

- 测试步骤必须详细具体，包含：
  - 操作对象（按钮名称、输入框标签）
  - 操作动作（点击、输入、选择）
  - 输入数据（具体值，如"zhangssan"，不能只写"用户名"）
- 使用编号列表组织步骤（1. 2. 3.）
- 步骤之间有清晰的逻辑顺序

#### 预期结果量化（强制）

- 禁止使用模糊词：❌ "正常显示" "成功" "合理"
- 使用具体表达：✅ "显示提示'用户创建成功'" "用户列表增加1条记录" "响应时间<2秒"
- 每个验证点单独列出（使用编号列表）
- UI变化要说明具体元素和状态

#### 场景覆盖完整性（强制）

每个功能至少包含以下测试场景：

1. **正常路径测试（P0）**: 覆盖核心业务流程
2. **参数验证测试（P1）**: 验证输入参数的格式、长度、类型等约束
3. **异常场景测试（P1）**: 验证错误提示和异常处理
4. **边界值测试（P1-P2）**: 验证最小值、最大值、临界值
5. **性能测试（P2）**: 验证响应时间、并发处理能力（如适用）
6. **安全测试（P1-P2）**: 验证权限控制、数据加密（如适用）

### 3. 格式规范

#### 文档格式

- 使用Markdown表格格式
- 表格列宽自适应，确保可读性
- 多行内容使用`<br>`换行或Markdown换行语法

#### 用例编号规则

**格式**: `{系统简称}-{功能模块简称}-{序号}`

**示例**:
- `UUV-USER-ADD-001`: UUV系统，用户管理-新增用户，第1个用例
- `UUV-USER-ADD-002`: UUV系统，用户管理-新增用户，第2个用例
- `UUV-LM-LC-001`: UUV系统，用户管理-账号检查，第1个用例

**编号原则**:
- 系统简称：2-5个字母，大写
- 功能模块简称：每个层级取2-4个字母，大写
- 序号：001开始，连续编号

#### 优先级分类

| 优先级 | 说明 | 适用场景 |
|--------|------|----------|
| **P0** 或 **高** | 核心流程，阻塞性功能 | 用户注册、登录、核心业务流程 |
| **P1** 或 **中** | 重要功能，影响用户体验 | 参数验证、异常处理、边界值测试 |
| **P2** 或 **低** | 辅助功能，优化类需求 | 性能优化、界面美化、非核心功能 |

#### 标签分类

**推荐标签**:
- `功能测试`: 正常业务流程测试
- `边界测试`: 边界值、临界值测试
- `异常测试`: 异常场景、错误处理测试
- `参数验证`: 输入参数格式、长度、类型验证
- `性能测试`: 响应时间、并发测试
- `安全测试`: 权限控制、数据加密测试
- `可自动化`: 可转化为E2E自动化测试的用例

**标签使用原则**:
- 每个用例至少1个标签，最多3个标签
- 使用逗号分隔多个标签
- "可自动化"标签用于标识可转化为E2E测试的用例

### 4. 特别说明

#### 测试步骤编写原则

**详细具体原则**:
- ✅ 正确: "点击【新增用户】按钮"
- ❌ 错误: "打开新增表单"

**操作可重复原则**:
- ✅ 正确: "1. 点击【新增用户】 2. 输入用户名'zhangssan' 3. 点击【保存】"
- ❌ 错误: "新增用户并保存"

**包含验证点原则**:
- ✅ 正确: "3. 点击【保存】，验证页面显示'保存成功'"
- ❌ 错误: "3. 点击【保存】"

#### E2E测试转化指引

**可自动化测试用例的特征**:

1. **UI操作清晰**: 所有操作对象有明确的标识（按钮名称、输入框ID）
2. **验证点明确**: 预期结果可以通过元素状态、文本内容验证
3. **无人工判断**: 不依赖人工判断的视觉效果（如"界面美观"）
4. **数据可准备**: 测试数据可以通过脚本或API预先准备

**E2E转化提示**:

在用例的"标签"字段标注`可自动化`，表示该用例可转化为Playwright等E2E测试脚本。

**E2E转化示例**:

```
用例编号: UUV-USER-ADD-001
标签: 功能测试,正常流程,可自动化
输入数据或操作:
1. 打开登录页面
2. 输入用户名"admin"，密码"123456"
3. 点击【登录】按钮
4. 验证跳转到首页
5. 点击【用户管理】菜单
6. 点击【新增用户】按钮

→ E2E转化:
await page.goto('/login');
await page.fill('#username', 'admin');
await page.fill('#password', '123456');
await page.click('button:has-text("登录")');
await expect(page).toHaveURL('/home');
await page.click('text=用户管理');
await page.click('button:has-text("新增用户")');
```

#### 信息不足的处理

如果输入材料中缺少某些信息，你应该：

1. **合理推断**: 基于行业经验和常见实践给出合理的测试用例
   - 示例：如果验收标准未明确错误提示语，可推断为"显示'XXX错误'提示"
   - 标注：`（推断，需与开发确认错误提示具体文案）`

2. **优先覆盖已知场景**: 先覆盖输入材料中明确的验收规则，再补充推断的测试场景

3. **列入待确认项**: 在测试用例的"测试空白"列标注待确认的信息
   - 示例：`【待确认】错误提示具体文案`

4. **保持保守**: 不确定的非功能需求（如性能指标）可以不生成测试用例，或生成P2优先级用例并标注"待确认"

#### 测试覆盖策略

**优先级分配原则**:

1. **P0用例（高优先级）**: 
   - 正常业务流程（Happy Path）
   - 核心功能验证
   - 数量：每个功能1-3个P0用例

2. **P1用例（中优先级）**:
   - 异常场景测试
   - 参数验证测试
   - 边界值测试
   - 数量：每个功能3-8个P1用例

3. **P2用例（低优先级）**:
   - 性能测试
   - 安全测试
   - 边缘场景测试
   - 数量：每个功能2-4个P2用例

**测试用例数量建议**:

- **简单功能**（如单一输入表单）：5-8个测试用例
- **中等功能**（如用户注册）：8-12个测试用例
- **复杂功能**（如订单流程）：12-20个测试用例
